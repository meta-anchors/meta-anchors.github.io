## Overview: 복잡성의 폭발과 추론 불가능한 영역

현대적인 Spring 기반 MSA 환경에서 우리는 데이터 일관성과 시스템 간 통합을 위해 다양한 도구를 결합합니다. **Spring Application Event, Domain Event, CDC(Kafka/Debezium)** 등이 바로 그것입니다. 

하지만 이 도구들이 한데 섞이는 순간, 시스템은 급속도로 복잡해지며 이론적으로도 완전한 해결이 불가능한 **'추론 불가능 영역'**으로 진입하게 됩니다. 본 포스트에서는 왜 이러한 경계 충돌이 발생하는지, 그리고 왜 이것이 설계의 미숙함이 아닌 구조적 필연성인지를 분석합니다.

---

## 이벤트 계층의 분류와 의미론적 차이

우리는 모두 '이벤트'라고 부르지만, 그 내면의 의미와 보장 수준은 완전히 다릅니다. 이들을 동일한 추상화로 다루는 것이 충돌의 시작입니다.

| Event 유형 | 의미 (Semantics) | 보장 수준 (Guarantees) |
| :--- | :--- | :--- |
| **Spring Event** | 특정 코드 블록의 실행 사실 | 보장 없음 (애플리케이션 내부 신호) |
| **Domain Event** | 비즈니스 로직상의 유의미한 변화 | 제한적 보장 (트랜잭션 전파에 의존) |
| **CDC Event** | DB 레코드의 물리적 상태 변화 | **강함** (DB Commit 완료 후 로그 기반) |

## 시간 경계(Time Bound)의 충돌

동일한 '사건'이 발생하더라도, 각 시스템이 인식하는 시간축은 서로 엇갈립니다.

### 세 가지 시간의 공존
- **Spring Transaction Time**: `begin` → `execute` → `commit`. Thread와 DB 중심의 명확한 스코프
- **Event Dispatch Time**: 비동기 여부에 따라 실행 시점이 결정되지 않음 (Scheduler 의존)
**CDC Log Time**: DB 커밋 이후 Kafka 등에 어펜드되는 시점. 가장 견고하지만 가장 늦음

이러한 시차로 인해, 관찰자는 동일한 사건을 **서로 다른 시간, 서로 다른 순서, 서로 다른 의미**로 관측하게 됩니다.

---

## 판단 위치의 역전 (Inversion of Judgment)

기존의 모노리스 방식에서는 애플리케이션 코드가 유일한 판단 주체였으며, 국소적 불변성을 유지하기 쉬웠습니다. 하지만 이벤트 기반 구조에서는 이 판단이 분산됩니다.

* **판단 복제**: N개의 리스너와 컨슈머가 각자의 시점에서 이벤트를 해석하고 판단합니다.
* **비용 증가**: 동일한 비즈니스 규칙을 여러 곳에서 중복 검증해야 하며, 판단 비용이 N배로 증가합니다.

---

## 분산 락(Distributed Lock)과 이벤트의 불협화음

분산 락은 상태를 보호하려 하지만, 이벤트 시스템과는 근본적으로 결이 맞지 않습니다.

* **Lock vs Transaction**: 락의 임대 시간(Lease Time)과 DB 트랜잭션의 생명 주기는 일치하지 않습니다.
* **CDC의 맹점**: CDC는 오직 DB의 최종 커밋 결과만 봅니다. 그 과정에서 어떤 락이 잡혔었는지, 어떤 경합이 있었는지는 관측 범위를 벗어납니다.

---

## 비동기 결합에 따른 상태 공간의 폭발

Async가 결합되면 실패 경로(Failure Path)는 인간의 인지 능력을 넘어섭니다.

* **시간 은닉**: 지연 실행과 재시도(Retry)가 반복되면서 "현재 시스템의 상태"를 정의하기 어려워집니다.
* **재현 불가능한 장애**: Spring Retry, Async Retry, Consumer Retry가 얽히며 발생하는 장애는 간헐적이며 로그만으로는 재현이 불가능합니다.

---

## 결론: 왜 해결이 어려운가?

이는 단순한 숙련도의 문제를 넘어, **경계 모델의 비합성성(Non-composability)** 때문입니다.

| 요소 | 중심 경계 모델 |
| :--- | :--- |
| **Transaction** | DB / Thread |
| **Event** | Dispatcher |
| **CDC** | Log Persistence |
| **Lock** | External Time (Redis/Zookeeper) |
| **Async** | Scheduler |

각 요소는 서로 다른 기준점을 가지고 있으며, 이를 하나로 통합할 공통 분모가 존재하지 않습니다. 
결과적으로 이벤트가 늘어날수록 시스템의 안정성은 하락하고 판단 비용은 상승합니다.

### 실무적 가이드라인
- **이벤트 계층 분리**: Spring/Domain/CDC 이벤트의 용도를 명확히 분리하고 혼용하지 마십시오
- **도메인 이벤트의 제한**: 명확한 비즈니스 단위 내에서만 사용하십시오
- **CDC의 역할 국한**: 데이터 통합 및 동기화 목적으로만 사용하고 비즈니스 트리거로 쓰지 마십시오
- **최소주의**: 분산 락과 비동기 처리는 최후의 수단으로만 사용하십시오
